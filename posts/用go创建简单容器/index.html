linux namespace

UTS namespace

UTS namespace常用来隔离子进程与父进程的hostname

package main

import (
	"fmt"
	"os"
	"os/exec"
	"syscall"
)

func main() {
	cmd := exec.Command("/bin/sh")

	cmd.Stdin = os.Stdin
	cmd.Stdout = os.Stdout
	cmd.Stderr = os.Stderr

	cmd.SysProcAttr = &syscall.SysProcAttr{
		// UTS namespace
		Cloneflags: syscall.CLONE_NEWUTS,
	}

	if err := cmd.Run(); err != nil {
		fmt.Printf("Error running the exec.Command - %s\n", err)
		os.Exit(1)
	}
}

PID namespace

PID = process id, 顾名思义，进程编号。父进程A创建了带PID namespace的子进程B，B会拥有和A一样的PID， 但是B只能操作自己PID namespace下的进程，不能操作A， 但是A能操作B。 为了节省篇幅，我用类似增量的形式来表达区别：

......
cmd.SysProcAttr = &syscall.SysProcAttr{
  // UTS namespace
  Cloneflags: syscall.CLONE_NEWPID,
}
......

继续编译执行

go build main.go
sudo ./main

在弹出的shell内执行

ps -ef

可以找到我们创建的父子进程

UID          PID    PPID  C STIME TTY          TIME CMD
......
root     2779448 2685255  0 20:40 pts/14   00:00:00 sudo ./main
root     2779449 2779448  0 20:40 pts/14   00:00:00 ./main
root     2779454 2779449  0 20:40 pts/14   00:00:00 /bin/sh
......

现在我们在新创建的PID namespace内的进程(PID=2779454)中，我们试着删除位于namespace外的父进程(PID=2779449)

kill -9 2779449

得到

/bin/sh: 3: kill: No such process

ps命令查看进程状态是利用linux特殊文件系统/proc实现的，所以只要没对这个进程做fs mount,linux内的进程都是可以看到所有进程状态的，这次响应no such process是因为PID namespace生效了，子进程看不到PID namespace外部的父进程。

UID namespace

如果需要将一个非root用户/组映射到某一个进程的root用户/组，就会用到UID namespace

......
cmd.SysProcAttr = &syscall.SysProcAttr{
  // UTS namespace
  Cloneflags: syscall.CLONE_NEWUSER,
  UidMappings: []syscall.SysProcIDMap{
    {
      ContainerID: 0,
      HostID: os.Getuid(),
      Size: 1,
    },
  },
  GidMappings: []syscall.SysProcIDMap{
    {
      ContainerID: 0,
      HostID: os.Getegid(),
      Size: 1,
    },
  },
}
......

在根目录创建文件需要root权限（通常sudo执行），我们尝试创建个文件：

# touch /testpermission
touch: 无法创建 '/testpermission': 权限不够

在这个进程中即使看起来是root用户在操作,但是在host中，还是以普通用户身份操作，这就是用户的映射。
